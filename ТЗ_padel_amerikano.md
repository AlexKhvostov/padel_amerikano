ТЗ: Веб-сервис «Падел Американо»
1. Общее описание

Веб-сервис для проведения любительских турниров по системе «Американо» в падел-теннисе.

Американо — турнирный формат, где:

Игроки постоянно меняют партнёров и соперников после каждого раунда
Очки считаются индивидуально для каждого игрока
Победитель определяется по наибольшей сумме личных очков

Ключевые требования:

Mobile-first интерфейс (приоритет экранов 360–430px)
Vanilla JS, без фреймворков
Бэкенд с REST API, MySQL база данных на сервере заказчика
Интерфейс полностью на русском языке
2. Технический стек
Слой	Технология
Frontend	Vanilla JS, HTML5, CSS3
Адаптивность	Mobile-first, медиазапросы
Backend	PHP или Node.js (на усмотрение разработчика)
База данных	MySQL
Хранение сессии	localStorage
Коммуникация	REST API (JSON)
3. Система «Компаний»
3.1 Концепция

«Компания» — постоянная группа людей с единым административным доступом, списком участников, историей турниров и общим рейтингом. Администратор входит по цифровому коду и получает доступ к управлению. Участники и зрители открывают отдельную непредсказуемую ссылку или QR-код компании и получают доступ к хабу турниров только для просмотра.

3.2 Создание компании
Главным экраном является компактный публичный список компаний; в его шапке расположена кнопка «Создать компанию»
Кнопка открывает модальное окно, в котором пользователь вводит название компании (например: «Клуб Ракетка», «Турнир у Димы»)
Название должно быть уникальным среди активных компаний без учёта регистра. После логического удаления компании её название можно использовать повторно
Система автоматически генерирует простой цифровой код администратора (4–6 цифр) и отдельный ключ просмотра
Компания сохраняется в БД
После создания в том же модальном окне крупно отображается:
Компания: Клуб Ракетка
Код администратора: 4872

Код администратора сохраняет организатор. Интерфейс предупреждает, что без кода управление восстановить нельзя, и предлагает отправить данные себе в Telegram. Кнопка «Поделиться просмотром» отправляет зрительскую ссылку без административного кода.
3.3 Вход в компанию
На главном экране — список и поле поиска по названию компании
Кнопка «Войти» в строке компании открывает компактное модальное окно ввода кода администратора
После успешного входа — переход в хаб компании со списком турниров
Сессия сохраняется в localStorage: при перезагрузке страницы пользователь остаётся авторизованным
Зрительская ссылка открывает хаб компании с текущим и завершёнными турнирами. В этом режиме нельзя менять участников, настройки, раунды и счёт.
3.4 Выход
У администратора кнопка «Выйти из компании» находится только внизу настроек и оформлена заметной строкой с пиктограммой выхода
Очищает localStorage, возвращает на главный экран
В режиме зрителя нижнее меню содержит кнопку «Выйти из просмотра», которая открывает публичный список компаний
3.5 Публичный список компаний
Публичный список компаний является главным экраном сервиса
Показываются активные, не удалённые компании по 10 записей на страницу
Над списком расположены вкладки «Мои» и «Все». Во вкладке «Мои» сначала показываются компании, отмеченные звездой, затем до 30 последних компаний, в которые входили администратором или зрителем на этом устройстве
Избранное и история посещений хранятся локально в браузере без регистрации, не удаляются при выходе из компании и не синхронизируются между устройствами
Если локальный список пуст, при открытии сервиса автоматически выбирается вкладка «Все»
Список сортируется по последнему обновлению компании: добавлению участника или турнира либо изменению счёта
Доступен поиск по названию внутри активной вкладки; фильтр по дате отсутствует
Для каждой компании выводятся количество участников, турниров, сыгранных и запланированных матчей, последнее обновление и отметка текущего турнира
Информационный статус «Турнир идёт» показывается, только если активный турнир обновлялся не более 15 минут назад. После 15 минут без новых раундов или результатов показывается «Заброшен», но турнир не останавливается и может быть продолжен
Для созданного, но ещё не начатого турнира в течение часа после последнего изменения показывается «Собирается турнир»
Фильтр статуса позволяет показать все компании либо только компании со статусом «Турнир идёт», «Собирается» или «Заброшен»
При прокрутке списка закрепляются шапка «Американо» с кнопкой создания, вкладки, фильтр статуса и поиск; карточки компаний прокручиваются под ними
Кнопка «Смотреть» открывает зрительский хаб компании
Кнопка «Войти» открывает модальное окно административного входа с уже выбранной компанией
3.6 Турниры компании
Администратор может создавать несколько турниров и просматривать историю завершённых игр
Турнир содержит собственные название, дату, количество кортов, состав, раунды, матчи и рейтинг
Состав турнира выбирается из активных участников компании и содержит от 4 до 36 игроков
Состав, название и корты можно менять до запуска первого раунда
Одновременно в компании может идти только один турнир; создавать будущие черновики разрешено
Завершённые турниры сохраняются и не могут быть сброшены; отдельный счёт можно исправить, после чего рейтинги пересчитываются
Администратор может удалить турнир только до запуска первого раунда. Действие требует подтверждения и необратимо; начатые и завершённые турниры удалить нельзя
4. Настройки турнира

Отдельный экран настроек турнира открывается пиктограммой шестерёнки в заголовке раундов.

4.1 Параметры
Параметр	Тип	Описание
Количество кортов	Число	Количество одновременно задействованных кортов
4.2 Логика счёта
Организатор свободно вводит фактический счёт матча
Каждый игрок получает количество очков, набранное его командой
После создания турнира открывается компактное окно правил. Оно также доступно по кнопке «i» в заголовке раундов
Основной рекомендуемый формат — `16+1`: каждый из четырёх игроков выполняет по 4 подачи, затем всегда играется 17-я подача первым подававшим игроком; принимающая пара выбирает принимающего
Альтернативный формат — `24+1`: по 6 подач каждого игрока и, если организаторы выбрали одинаковую длину всех матчей, дополнительный 25-й мяч по тому же правилу
Один выбранный формат используется во всех матчах турнира
Суммы очков 16, 17, 24 и 25 считаются стандартными и сохраняются без предупреждения
При другой сумме система показывает предупреждение и сохраняет счёт только после явного подтверждения организатора
Оба значения должны быть целыми неотрицательными числами
Ничья не допускается: значения команд должны отличаться
4.3 Изменение настроек
Настройки можно менять до начала первого раунда
После начала турнира — настройки заблокированы с предупреждением: «Турнир начат. Изменение настроек недоступно»
Кнопка «Сбросить турнир» доступна только для незавершённого турнира, удаляет его раунды и разблокирует настройки (с подтверждением)
Маленькая кнопка «Удалить компанию» содержит только пиктограмму корзины и расположена рядом с выходом внизу настроек. Она требует подтверждения и выполняет логическое удаление: данные остаются в БД, но компания исчезает из поиска, публичного списка и становится недоступна по прежним ссылкам и сессиям
4.4 Управление компанией
Администратор может переименовать компанию в настройках в любой момент
Новое название проверяется на уникальность среди активных компаний
Изменение названия и изменение кода представлены рядом как два равнозначных действия с понятными пиктограммами; каждое открывает собственную компактную форму
В настройках отображается скрытый код администратора с кнопкой показа
Администратор может изменить код, указав текущий код и дважды введя новый цифровой код длиной от 4 до 8 цифр
В настройках отображаются короткая зрительская ссылка вида `/v/{slug}`, QR-код и кнопка «Поделиться просмотром»
5. Список игроков
5.1 Добавление игрока

Форма добавления содержит поля:

Имя (обязательное) — текстовое поле
Telegram (необязательное) — ник вида @username или полная ссылка https://t.me/username
5.2 Отображение списка

Каждый игрок отображается компактной строкой:

[Порядковый номер]  Иван Петров
                    [✏️ Редактировать]  [🗑️ Удалить]

Telegram не загромождает общий список и отображается как кликабельная ссылка только в модальном окне подробной статистики
По умолчанию список показывает только активных участников. Компактный переключатель «Активные / Все» расположен в строке заголовка и не увеличивает её высоту
Если ник указан без @ — добавлять автоматически
Если указана полная ссылка — отображать как @username
5.3 Ограничения
Минимум 4 игрока для старта турнира
Максимум 200 активных участников в компании; в одном турнире выбирается не более 36
Нельзя добавить двух игроков с одинаковым именем в одной компании
Если игрок удаляется после начала турнира — его исторические результаты сохраняются, но он помечается как «неактивный»
В модальном окне неактивного игрока администратору доступна кнопка «Активировать игрока». Повторная кнопка удаления для неактивного игрока не показывается
5.4 Счётчик

В верхней части экрана показывается число активных участников компании. Список одновременно является общим рейтингом: в карточке участника отображаются доля набранных очков, средняя разница очков за матч, процент побед, количество игр и турниров
По нажатию на карточку по центру экрана открывается подробная статистика и история выступлений по турнирам с местом, очками и количеством игр. Окно закрывается крестиком, клавишей Escape или нажатием на затемнённую область вне карточки

6. Генерация раундов
6.1 Алгоритм ротации
Полное расписание формируется автоматически при старте турнира
Цель полной ротации: каждый игрок играет в паре с каждым другим игроком ровно один раз, когда это математически возможно
Если число всех уникальных партнёрских пар нечётное (N mod 4 = 2 или 3), допускается один минимальный повтор партнёрства; два игрока сыграют на один матч больше
Соперники распределяются максимально равномерно: система минимизирует разницу в количестве встреч между любыми двумя игроками
Один раунд — один временной слот
В одном раунде каждый игрок может участвовать не более чем в одном матче
1 корт = 1 одновременный матч = 2 пары (4 игрока)
В раунде проводится не больше матчей, чем задано кортов
Плановые будущие раунды публикуются по одному после завершения текущего
6.2 Нечётное количество игроков (не кратно 4)
Если игроков больше, чем помещается на кортах, остальные находятся на «скамейке»
Скамейка распределяется равномерно: разница в количестве пропущенных раундов между активными игроками должна быть минимальной
Правило применяется для любого остатка и при недостаточном количестве кортов
Для N mod 4 = 2 или 3 система предупреждает, что для полного покрытия потребуется один повтор партнёрства

Игроки на «скамейке» чётко отображаются в раунде: «Пропускают раунд: Антон, Мария»

6.3 Количество раундов
Количество раундов рассчитывается по фактически построенному полному расписанию
Для N, кратного 4, и достаточного количества кортов полная ротация занимает N − 1 раундов
При меньшем количестве кортов или составе, не кратном 4, расписание может содержать больше раундов
До старта система показывает количество раундов, матчей и ожидаемое число игр каждого участника
Организатор публикует следующий плановый раунд кнопкой «▶ Следующий раунд»
Раунды нумеруются по порядку: Раунд 1, Раунд 2...
7. Экран «Раунды»
7.1 Структура экрана
Список раундов в виде аккордеона или табов
Текущий активный раунд раскрыт по умолчанию
Завершённые раунды можно свернуть
Кнопка «Показать всю сетку» открывает компактное модальное окно со всеми запланированными раундами, матчами и распределением по кортам
Для заметности кнопка примерно раз в 13 секунд показывает короткий блик и плавно превращает пиктограмму сетки в теннисный мяч с контуром и характерными швами, задерживает его примерно на секунду и превращает обратно; при системном ограничении анимации эффект отключается
Имена двух игроков одной команды отображаются в две строки, чтобы длинные имена не ломали мобильную вёрстку
Состояние раундов и счёта автоматически обновляется у зрителей без перезагрузки страницы
7.2 Отображение матча
┌─────────────────────────────────────────┐
│  КОРТ 1                                 │
│                                         │
│  🟦 Иван + Сергей                       │
│  vs                                     │
│  🟥 Олег + Дмитрий                      │
│                                         │
│  [ 8 ]  :  [ 8 ]   ← поля ввода счёта  │
│                                         │
│  [     Сохранить     ]                  │
└─────────────────────────────────────────┘

7.3 Ввод счёта
Два числовых поля — одно для каждой команды
Валидация при сохранении:
Оба поля заполнены
Оба значения — целые неотрицательные числа
Разница между командами ≥ 1 (ничьей не бывает)
При сумме 16, 17, 24 или 25 счёт сохраняется сразу
При другой сумме показывается предупреждение с подтверждением сохранения
При корректном вводе — матч помечается зелёной галочкой ✅
Уже сохранённый счёт можно редактировать (кнопка «✏️»)
7.4 Завершение раунда
Когда все матчи раунда завершены — появляется кнопка «▶ Следующий раунд»
При нажатии генерируется следующий раунд с новой ротацией
Генерация учитывает историю предыдущих пар
8. Экран «Рейтинг»
Рейтинг выбранного турнира отображается отдельным экраном. Общий рейтинг компании встроен в список участников, чтобы не дублировать навигацию
8.1 Таблица рейтинга
#	Игрок	Матчей	Очков	Побед	Поражений
🥇 1	Иван	6	94	5	1
🥈 2	Сергей	6	88	4	2
🥉 3	Олег	6	81	3	3
8.2 Правила подсчёта
Очки — сумма всех очков игрока за все матчи (очки своей команды в каждом матче)
Победа — команда игрока набрала больше очков в матче
Поражение — команда игрока набрала меньше очков в матче
Доля очков — среднее по матчам отношение очков команды игрока ко всем очкам матча
Средняя разница — среднее значение «очки команды игрока минус очки соперников» за матч
В подробной статистике итоговые места всех завершённых турниров объединяются в наглядную шкалу: слева первое место, справа последнее. Маркер показывает, где игрок в среднем располагается в турнирных сетках; непонятное процентное значение пользователю не выводится
8.3 Сортировка
В отдельном турнире — по сумме очков (по убыванию)
При равных очках — по победам (по убыванию)
При равных очках и победах — по алфавиту
В списке компании игроки с 5 и более матчами располагаются выше предварительных рейтингов, затем сортируются по доле набранных очков, средней разнице, проценту побед, количеству матчей и имени. Неактивные игроки располагаются после активных
8.4 Обновление
Рейтинг обновляется в реальном времени после каждого сохранённого счёта
Каждый игрок видит своё текущее место
Для каждого игрока компактно отображаются сыгранные и запланированные матчи, доля очков, процент побед и средняя разница
До 5 сыгранных матчей рейтинг помечается как предварительный. Общая сумма очков и абсолютное число побед остаются только в подробной статистике и не используются как критерии общего рейтинга
9. Структура базы данных (MySQL)
Полная актуальная схема находится в `sql/schema.sql`. Основные таблицы и ограничения:

`companies` — название, bcrypt-хеш кода администратора, уникальные `view_token` и `view_slug`, JSON-настройки, дата создания и `deleted_at`. Виртуальное поле `active_name` с уникальным индексом запрещает одинаковые названия активных компаний и позволяет повторно использовать название после логического удаления.

`players` — компания, имя, Telegram, активность и дата создания. Сочетание компании и имени игрока уникально.

`tournaments` — компания, название, статус `draft / active / completed`, JSON-настройки, даты создания, изменения, старта и завершения. Уникальное ограничение допускает только один активный турнир компании. «Собирается» и «Заброшен» являются вычисляемыми информационными статусами и не меняют жизненный цикл автоматически.

`tournament_players` — выбранное подмножество участников компании для отдельного турнира.

`rounds` — компания, турнир, номер, JSON со скамейкой, статус `planned / active / completed` и дата создания. Нумерация раундов уникальна внутри турнира; для компании одновременно допускается только один активный раунд.

`matches` — раунд и номер корта.

`match_players` — игрок, матч и команда 1 или 2. Один игрок не может быть добавлен в матч дважды.

`match_scores` — уникальный матч, очки обеих команд, признак завершения и время обновления.

`login_attempts` — IP-адрес, нормализованное название компании и время неудачной попытки входа; используется для защиты от перебора.

Все связанные сущности изолированы по компании и турниру. Удаление компании выполняется логически, а не каскадным физическим удалением из интерфейса.

11. Навигация

Нижнее меню уровня компании для администратора: «Участники / Турниры / Настройки». Для зрителя: «Участники / Турниры / Выйти».

Основное меню «Участники / Турниры / Настройки» не меняется при входе в турнир, а «Турниры» остаётся активным родительским разделом. Над ним появляется визуально связанная компактная панель второго уровня «Раунды / Рейтинг». Нажатие основной кнопки «Турниры» возвращает в хаб; настройки турнира открываются шестерёнкой в заголовке раундов.

Хаб турниров показывает активный, собирающийся, заброшенный, запланированные и завершённые турниры со статусом, составом и прогрессом матчей. После входа администратор и зритель сначала попадают в этот хаб.

12. UX / UI требования
12.1 Mobile-first
Основной целевой экран: 360px – 430px ширины
Компактная мобильная шкала: основные кнопки — 44px, поля ввода — 40px, встроенные действия и иконки — 36px; элементы должны сохранять удобную область касания
Числовые поля счёта открывают цифровую клавиатуру (inputmode="numeric")
Нет горизонтального скролла
Текст в полях ввода — не менее 16px, чтобы мобильные браузеры не увеличивали страницу при фокусе
12.2 Общие принципы
Современная светлая палитра: нейтральные серо-голубые поверхности, графитовый текст и сине-фиолетовый акцент; зелёный не используется как основной цвет интерфейса
Чёткая визуальная иерархия: название матча, команды, поля счёта
Цветовое различие команд (синяя / красная метка)
Завершённые матчи и раунды визуально отличаются от активных
Все деструктивные действия (удалить, сбросить) требуют подтверждения
12.3 Состояния загрузки
При каждом запросе к API — индикатор загрузки (спиннер)
При ошибке сети — понятное сообщение пользователю с кнопкой «Повторить»
12.4 Уведомления
Toast-уведомления (всплывающие снизу) при:
Успешном сохранении счёта
Добавлении игрока
Ошибке валидации
13. Безопасность
Код администратора проверяется по bcrypt-хешу; токен просмотра даёт только права чтения
Все запросы к API проверяют принадлежность данных к компании и требуемую роль
Защита от SQL-инъекций — использовать параметризованные запросы (PDO / prepared statements)
Защита от брутфорса: после 3 ошибок входа для одной компании с одного IP доступ блокируется на 10 минут
После 20 неудачных попыток с одного IP по разным компаниям вход с этого IP блокируется на 10 минут
Успешный вход очищает накопленные ошибки для сочетания IP и компании
Форма входа возвращает одинаковую ошибку для неизвестной компании и неправильного кода, не раскрывая существование учётной записи
В production обязательно используются HTTPS, `APP_DEBUG=false`, секрет подписи токенов длиной не менее 32 символов и корректная настройка доверенных reverse proxy
