ТЗ: Веб-сервис «Падел Американо»
1. Общее описание

Веб-сервис для проведения любительских турниров по системе «Американо» в падел-теннисе.

Американо — турнирный формат, где:

Игроки постоянно меняют партнёров и соперников после каждого раунда
Очки считаются индивидуально для каждого игрока
Победитель определяется по наибольшей сумме личных очков

Ключевые требования:

Mobile-first интерфейс (приоритет экранов 360–430px)
Vanilla JS, без фреймворков
Бэкенд с REST API, MySQL база данных на сервере заказчика
Интерфейс полностью на русском языке
2. Технический стек
Слой	Технология
Frontend	Vanilla JS, HTML5, CSS3
Адаптивность	Mobile-first, медиазапросы

3. Система «Компаний»
3.1 Концепция

«Компания» — это изолированное пространство одного турнира или серии турниров. Администратор входит по цифровому коду и получает доступ к управлению. Участники и зрители открывают отдельную непредсказуемую ссылку или QR-код и получают доступ только для просмотра.

3.2 Создание компании
Главный экран содержит кнопку «Создать компанию»
Пользователь вводит название компании (например: «Клуб Ракетка», «Турнир у Димы»)
Название должно быть уникальным среди активных компаний без учёта регистра. После логического удаления компании её название можно использовать повторно
Система автоматически генерирует простой цифровой код администратора (4–6 цифр) и отдельный ключ просмотра
Компания сохраняется в БД
После создания на экране крупно отображается:
Компания: Клуб Ракетка
Код администратора: 4872

Код администратора сохраняет организатор. Интерфейс предупреждает, что без кода управление восстановить нельзя, и предлагает отправить данные себе в Telegram. Кнопка «Поделиться просмотром» отправляет зрительскую ссылку без административного кода.
3.3 Вход в компанию
На главном экране — поле поиска по названию компании
После выбора компании — поле ввода кода администратора
После успешного входа — переход в интерфейс управления турниром
Сессия сохраняется в localStorage: при перезагрузке страницы пользователь остаётся авторизованным
Зрительская ссылка сразу открывает турнир в режиме просмотра. В этом режиме нельзя менять игроков, настройки, раунды и счёт.
3.4 Выход
Кнопка «Выйти из компании» в настройках
Очищает localStorage, возвращает на главный экран
В режиме зрителя нижнее меню содержит кнопку «Выйти из просмотра», которая открывает публичный список игр
3.5 Публичный список игр
На главном экране доступна кнопка «Посмотреть все игры»
В список попадают все созданные компании; до первой ротации они имеют статус «Не начат»
Датой и временем старта турнира считается момент запуска первой ротации
По умолчанию показываются компании, созданные сегодня; доступен выбор даты и показ всех дат
Список группируется и сортируется по дате создания компании
Для каждой компании отдельно выводятся дата создания, старт турнира, статус, последнее обновление, участники и матчи «сыграно / всего»
Кнопка «Смотреть» открывает турнир с правами только для чтения
Кнопка «Войти» открывает форму административного входа с уже выбранной компанией
4. Настройки турнира

Отдельный экран «⚙️ Настройки» доступен из нижнего меню.

4.1 Параметры
Параметр	Тип	Описание
Количество кортов	Число	Количество одновременно задействованных кортов
4.2 Логика счёта
Организатор свободно вводит фактический счёт матча
Каждый игрок получает количество очков, набранное его командой
Суммы очков 16, 17, 24 и 25 считаются стандартными и сохраняются без предупреждения
При другой сумме система показывает предупреждение и сохраняет счёт только после явного подтверждения организатора
Оба значения должны быть целыми неотрицательными числами
Ничья не допускается: значения команд должны отличаться
4.3 Изменение настроек
Настройки можно менять до начала первого раунда
После начала турнира — настройки заблокированы с предупреждением: «Турнир начат. Изменение настроек недоступно»
Кнопка «Сбросить турнир» сбрасывает все раунды и разблокирует настройки (с подтверждением)
Кнопка «Удалить компанию» требует подтверждения и выполняет логическое удаление: данные остаются в БД, но компания исчезает из поиска, публичного списка и становится недоступна по прежним ссылкам и сессиям
4.4 Управление компанией
Администратор может переименовать компанию в настройках в любой момент
Новое название проверяется на уникальность среди активных компаний
В настройках отображается скрытый код администратора с кнопкой показа
В настройках отображаются короткая зрительская ссылка вида `/v/{slug}`, QR-код и кнопка «Поделиться просмотром»
5. Список игроков
5.1 Добавление игрока

Форма добавления содержит поля:

Имя (обязательное) — текстовое поле
Telegram (необязательное) — ник вида @username или полная ссылка https://t.me/username
5.2 Отображение списка

Каждый игрок отображается компактной строкой:

[Порядковый номер]  Иван Петров
                    @ivanpetrov  (кликабельная ссылка)
                    [✏️ Редактировать]  [🗑️ Удалить]

Telegram отображается как кликабельная ссылка, открывающая Telegram
Если ник указан без @ — добавлять автоматически
Если указана полная ссылка — отображать как @username
5.3 Ограничения
Минимум 4 игрока для старта турнира
Максимум 36 игроков
Нельзя добавить двух игроков с одинаковым именем в одной компании
Если игрок удаляется после начала турнира — его исторические результаты сохраняются, но он помечается как «неактивный»
5.4 Счётчик

В верхней части экрана показывается: «Игроков: 12 / 36»

6. Генерация раундов
6.1 Алгоритм ротации
Полное расписание формируется автоматически при старте турнира
Цель полной ротации: каждый игрок играет в паре с каждым другим игроком ровно один раз, когда это математически возможно
Если число всех уникальных партнёрских пар нечётное (N mod 4 = 2 или 3), допускается один минимальный повтор партнёрства; два игрока сыграют на один матч больше
Соперники распределяются максимально равномерно: система минимизирует разницу в количестве встреч между любыми двумя игроками
Один раунд — один временной слот
В одном раунде каждый игрок может участвовать не более чем в одном матче
1 корт = 1 одновременный матч = 2 пары (4 игрока)
В раунде проводится не больше матчей, чем задано кортов
Плановые будущие раунды публикуются по одному после завершения текущего
6.2 Нечётное количество игроков (не кратно 4)
Если игроков больше, чем помещается на кортах, остальные находятся на «скамейке»
Скамейка распределяется равномерно: разница в количестве пропущенных раундов между активными игроками должна быть минимальной
Правило применяется для любого остатка и при недостаточном количестве кортов
Для N mod 4 = 2 или 3 система предупреждает, что для полного покрытия потребуется один повтор партнёрства

Игроки на «скамейке» чётко отображаются в раунде: «Пропускают раунд: Антон, Мария»

6.3 Количество раундов
Количество раундов рассчитывается по фактически построенному полному расписанию
Для N, кратного 4, и достаточного количества кортов полная ротация занимает N − 1 раундов
При меньшем количестве кортов или составе, не кратном 4, расписание может содержать больше раундов
До старта система показывает количество раундов, матчей и ожидаемое число игр каждого участника
Организатор публикует следующий плановый раунд кнопкой «▶ Следующий раунд»
Раунды нумеруются по порядку: Раунд 1, Раунд 2...
7. Экран «Раунды»
7.1 Структура экрана
Список раундов в виде аккордеона или табов
Текущий активный раунд раскрыт по умолчанию
Завершённые раунды можно свернуть
Кнопка «Показать всю сетку» открывает компактное модальное окно со всеми запланированными раундами, матчами и распределением по кортам
Для заметности кнопка примерно раз в 13 секунд показывает короткий блик и плавно превращает пиктограмму сетки в теннисный мяч с контуром и характерными швами, задерживает его примерно на секунду и превращает обратно; при системном ограничении анимации эффект отключается
Имена двух игроков одной команды отображаются в две строки, чтобы длинные имена не ломали мобильную вёрстку
Состояние раундов и счёта автоматически обновляется у зрителей без перезагрузки страницы

7.3 Ввод счёта
Два числовых поля — одно для каждой команды
Валидация при сохранении:
Оба поля заполнены
Оба значения — целые неотрицательные числа
Разница между командами ≥ 1 (ничьей не бывает)
При сумме 16, 17, 24 или 25 счёт сохраняется сразу
При другой сумме показывается предупреждение с подтверждением сохранения
При корректном вводе — матч помечается зелёной галочкой ✅
Уже сохранённый счёт можно редактировать (кнопка «✏️»)
7.4 Завершение раунда
Когда все матчи раунда завершены — появляется кнопка «▶ Следующий раунд»
При нажатии генерируется следующий раунд с новой ротацией
Генерация учитывает историю предыдущих пар
8. Экран «Рейтинг»
8.1 Таблица рейтинга
#	Игрок	Telegram	Матчей	Очков	Побед	Поражений
🥇 1	Иван	@ivan	6	94	5	1
🥈 2	Сергей	@sergey	6	88	4	2
🥉 3	Олег	—	6	81	3	3
8.2 Правила подсчёта
Очки — сумма всех очков игрока за все матчи (очки своей команды в каждом матче)
Победа — команда игрока набрала больше очков в матче
Поражение — команда игрока набрала меньше очков в матче
8.3 Сортировка
По очкам (по убыванию)
При равных очках — по победам (по убыванию)
При равных очках и победах — по алфавиту
8.4 Обновление
Рейтинг обновляется в реальном времени после каждого сохранённого счёта
Каждый игрок видит своё текущее место
Для каждого игрока компактно отображаются сыгранные и запланированные матчи, победы и поражения
9. Структура базы данных (MySQL)
Полная актуальная схема находится в `sql/schema.sql`. Основные таблицы и ограничения:

`companies` — название, bcrypt-хеш кода администратора, уникальные `view_token` и `view_slug`, JSON-настройки, дата создания и `deleted_at`. Виртуальное поле `active_name` с уникальным индексом запрещает одинаковые названия активных компаний и позволяет повторно использовать название после логического удаления.

`players` — компания, имя, Telegram, активность и дата создания. Сочетание компании и имени игрока уникально.

`rounds` — компания, номер, JSON со скамейкой, статус `planned / active / completed` и дата создания. Для компании одновременно допускается только один активный раунд.

`matches` — раунд и номер корта.

`match_players` — игрок, матч и команда 1 или 2. Один игрок не может быть добавлен в матч дважды.

`match_scores` — уникальный матч, очки обеих команд, признак завершения и время обновления.

`login_attempts` — IP-адрес, нормализованное название компании и время неудачной попытки входа; используется для защиты от перебора.

Все связанные сущности изолированы по компании. Удаление компании выполняется логически, а не каскадным физическим удалением из интерфейса.

12. UX / UI требования
12.1 Mobile-first
Основной целевой экран: 360px – 430px ширины
Компактная мобильная шкала: основные кнопки — 44px, поля ввода — 40px, встроенные действия и иконки — 36px; элементы должны сохранять удобную область касания
Числовые поля счёта открывают цифровую клавиатуру (inputmode="numeric")
Нет горизонтального скролла
Текст в полях ввода — не менее 16px, чтобы мобильные браузеры не увеличивали страницу при фокусе
12.2 Общие принципы
Чёткая визуальная иерархия: название матча, команды, поля счёта
Цветовое различие команд (синяя / красная метка)
Завершённые матчи и раунды визуально отличаются от активных
Все деструктивные действия (удалить, сбросить) требуют подтверждения
12.3 Состояния загрузки
При каждом запросе к API — индикатор загрузки (спиннер)
При ошибке сети — понятное сообщение пользователю с кнопкой «Повторить»
12.4 Уведомления
Toast-уведомления (всплывающие снизу) при:
Успешном сохранении счёта
Добавлении игрока
Ошибке валидации
13. Безопасность
Код администратора проверяется по bcrypt-хешу; токен просмотра даёт только права чтения
Все запросы к API проверяют принадлежность данных к компании и требуемую роль
Защита от SQL-инъекций — использовать параметризованные запросы (PDO / prepared statements)
Защита от брутфорса: после 3 ошибок входа для одной компании с одного IP доступ блокируется на 10 минут
После 20 неудачных попыток с одного IP по разным компаниям вход с этого IP блокируется на 10 минут
Успешный вход очищает накопленные ошибки для сочетания IP и компании
Форма входа возвращает одинаковую ошибку для неизвестной компании и неправильного кода, не раскрывая существование учётной записи
В production обязательно используются HTTPS, `APP_DEBUG=false`, секрет подписи токенов длиной не менее 32 символов и корректная настройка доверенных reverse proxy
